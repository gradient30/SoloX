# SoloX 性能验收方案设计

**日期：** 2026-06-13

**范围：** 方案 A — 纯代码层性能验收，无需真机。

## 目标

- 减少报告相关 API 上可避免的服务端 CPU、文件 I/O 与内存开销。
- 保持当前工作区基线的路由、响应字段、状态码、图表顺序、默认 1500 点上限及原始 log 不变。
- 每项优化均用回归测试与优化前后对比数据证明有效。
- 不引入缓存、数据库、后台任务或新的运行时依赖。

## 已确认热点

1. `/apm/log` 构造字典时，各值会**立即调用**全部指标加载函数。请求单一指标时仍会读取所有指标文件（含 memory detail、battery、disk、FPS、network、GPU、CPU 核等）。
2. 分析页路由会**两次**枚举报告目录：一次用于对比选择器，一次检查所选报告是否存在。
3. 报告列表先枚举名称再单独取元数据；可在单次 `scandir` 中获取目录项，同时保持排序与分页。

## 选定方案

采用保守、局部的优化：

- `/apm/log` 仅调度**一个**请求的指标加载函数。
- 保留对比选择器的目录扫描，将重复的「是否存在」扫描改为**直接目录检查**。
- 报告列表使用 `os.scandir()`，尽早排除非目录项，并在同一次扫描中获取修改时间。

本轮**不改动**现有图表降采样算法与 API 载荷。流式降采样有意延后：要复现当前精确采样索引，需两次读文件或维护索引，复杂度高，且对常见报告体积可能反而增加延迟。

## 兼容性控制

- 测试断言：仅被选中的指标加载函数被调用。
- 测试断言：现有载荷元数据与未知 `target` 的错误行为不变。
- 测试断言：报告排序、分页与过滤不变。
- 测试断言：分析路由加载已有报告时不对根目录做第二次 `os.listdir()`。
- 每组变更后运行完整自动化回归套件。

## 验收标准

- `/apm/log?target=cpu` 仅调用 `getCpuLog`。
- API 响应结构保持兼容。
- 报告列表顺序与分页不变。
- 分析路由在查找所选报告时不再执行第二次 `os.listdir()`。
- 全量测试套件通过。
- 合成基准证明工作量下降且响应无回归。
