# SoloX 实时采集与分析性能设计

**日期：** 2026-06-13

**目标：** 在不改变指标口径、接口字段、报告原始数据和页面业务流程的前提下，降低实时采集轮询、分析页并发加载和大报告对比的资源开销，并提供可验收的运行时数据。

## 已确认问题

1. 实时页使用单一 `timerQ` 管理全部指标，并把函数调用结果传给 `setTimeout`。这会导致请求立即执行、指标之间互相清理定时器，无法形成稳定采样间隔。
2. `/apm/log/compare` 与 `/apm/log/pk` 未接入现有 `max_points` 降采样能力，大报告会全量读取并返回。
3. 分析页初始化时同时发起最多 9 个日志请求，增加磁盘突发读取、连接占用和页面主线程回调压力。
4. 当前缺少 API 请求耗时、并发峰值和 ADB 命令耗时统计，性能回归只能依赖主观观察。

## 设计

### 1. 实时采集调度

- 用按指标隔离的定时器注册表替代共享 `timerQ`。
- 下一次采集只在本次请求成功后排队，避免同一指标重叠请求。
- 页面不可见时不执行采集请求，仅保留低频调度检查；页面恢复后继续。
- 停止任务时统一清理全部指标定时器。

不改变现有采样 API、请求参数、成功回调和图表更新逻辑。

### 2. 对比与 PK 降采样

- `/apm/log/compare` 和 `/apm/log/pk` 接受可选 `max_points`。
- 显式正数参数由服务端封顶到 `CHART_DEFAULT_MAX_POINTS`；缺失、非法或非正数时保持旧 API 全量行为。
- `File` 对比读取方法增加可选参数并复用现有 `readLog(..., max_points)`。
- compare 与 PK 页面显式传递服务端提供的图表点数上限。

### 3. 分析页并发限制

- 页面引入轻量请求任务队列，最大并发数固定为 3。
- 各图表初始化函数返回原有 jqXHR，由队列在请求结束后启动下一项。
- 指标加载顺序保持 CPU、内存、网络、FPS、GPU、电池、可选明细的现有优先级。

### 4. 性能遥测

- API Blueprint 对 `/apm/` 请求记录当前并发、并发峰值、总次数、平均值、最大值和最近样本 P95。
- 统一 ADB `shell` 入口记录命令次数、活动数、峰值、平均值、最大值和 P95。
- `/apm/telemetry` 返回进程内快照；`reset=1` 可在验收前清零。
- API 响应增加 `X-SoloX-Response-Time-Ms` 诊断头，不改变 JSON 响应体。
- 样本使用有界队列，避免长期运行导致内存增长。

## 风险控制

- 所有新参数均为可选，旧客户端不传参时保持原有全量返回行为。
- 遥测仅使用标准库锁和有界内存，不依赖数据库或外部服务。
- 不调整 ADB 命令内容、指标公式、文件格式和报告目录结构。
- 不在本轮引入 Gunicorn、多 worker、WebSocket 或日志流式解析。

## 验收标准

- 实时页不存在 `setTimeout(getXxx(...))` 立即调用模式；不同指标使用独立 timer key。
- 停止任务可清理全部采集 timer；页面隐藏时不继续请求。
- compare/pk 页面和 API 显式 `max_points` 请求应用降采样，返回点数不超过指定上限；旧客户端缺省请求保持兼容。
- 分析页日志请求并发上限为 3，所有图表任务最终执行。
- 遥测接口能反映 API 与 ADB 调用统计，重置功能有效。
- 专项测试、完整测试、兼容性门禁和浏览器验收全部通过。
