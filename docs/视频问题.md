# SoloX 录屏 / 播放问题复盘（交接文档）

> 文档目的：供后续开发者快速定位录屏保存失败、画质差、播放器显示异常等问题。  
> 环境：Windows 10、Python 3.12、SoloX 开发服 `127.0.0.1:50005`、设备 `ecc3b00e(V1936A)`、测试游戏 `com.lyjz.chqsy.vivo`（Cocos2d / SurfaceView）。  
> 最后更新：2026-06-14（第三次补充：采用 scrcpy MKV + ffmpeg remux MP4）

## 问题总览

| # | 现象 | 根因类别 | 当前状态（2026-06-14） |
|---|------|----------|------------------------|
| 1 | MP4 保存后损坏、无播放按钮、系统播放器报 `0xc00d36c4` | Windows 上 scrcpy 停止方式不正确 → 未写入 `moov` | **已改方案**：scrcpy 先录 MKV，再用 ffmpeg `-c copy` 无损封装 MP4；MP4 校验通过才网页播放，待用户网页全流程手测 |
| 2 | 画质极差，不像 480p/720p | 录制后端选错（`screenrecord` 无法录 SurfaceView 游戏）+ 码率/编码器偏低 | 保持 scrcpy 录制 SurfaceView；remux 不重编码，避免二次损失；清晰度仍需真机手测 |
| 3 | 播放器中间只有一小块视频 | 前端 CSS：竖屏视频 + `max-height` 限制导致显示宽度极窄；或源视频本身只有中间有画面（screenrecord 问题） | 已改 `report.html` 布局；若源文件正常应铺满弹窗 |
| 4 | MKV 能保存并播放（但浏览器体验差） | MKV 头在文件开头，强杀后仍可能「有文件」；浏览器对 MKV 的 `duration`/seek 支持弱 | 当前 MKV 仅作为中间文件/系统播放器兜底；浏览器只播放 remux 后校验通过的 MP4 |

### 2026-06-14 第三次补充：当前采用的稳定化方案

| 项 | 当前实现 |
|----|----------|
| 录制容器 | scrcpy 输出 `record.mkv`，显式传 `--record-format=mkv` |
| 网页播放文件 | 停止后用 ffmpeg `-c copy -movflags +faststart` 生成 `record.mp4` |
| 播放按钮门槛 | 只有 `_is_valid_record_file(record.mp4, 'mp4')` 通过才视为网页可播 |
| MKV 角色 | 仅作为系统播放器兜底，`browser_playable=False` |
| ffmpeg 来源 | 优先 `SOLOX_FFMPEG`，其次 `solox/public/ffmpeg/` 内置目录，再查 PATH |
| ffmpeg 缺失 | `/apm/record/start` 直接失败并提示安装/配置，不再静默走不稳定 MP4 直录 |

| # | 尝试方案 | 改动要点 | 用户/日志反馈 | 结论 |
|---|----------|----------|---------------|------|
| 1.1 | 停止后轮询 `wait_for_record_file()`，写 `detect_record_video_flag()` | `common.py` `File` | 文件仍无 `moov` 时轮询超时 | ⚠️ 必要但不充分：不能代替正确 stop |
| 1.2 | `make_report()` 移动文件后再次检测并回写 `result.json` 的 `video` | `common.py` | 损坏文件仍为 `video:0` | ⚠️ 校验逻辑正确，无法「救活」坏文件 |
| 1.3 | Windows `CREATE_NEW_PROCESS_GROUP` + `os.kill(CTRL_BREAK_EVENT)` | `Scrcpy._graceful_stop_process` | 日志写 **「video finalized」** 但 30s 内仍 **无 moov**；用户：**无播放按钮、MP4 损坏 0xc00d36c4** | ❌ **误判成功**：进程退出 ≠ 文件 finalize |
| 1.4 | 修正日志文案为「exited after stop signal」 | 同上 | 仍无 moov，但不再误导 | ✅ 应保留 honest logging |
| 1.5 | `AttachConsole` + `GenerateConsoleCtrlEvent(0,0)`（早期，配合 CREATE_NEW_PROCESS_GROUP） | `_windows_send_ctrl_event` | 与 1.3 类似，仍失败 | ❌ 进程组/console 组合不对时无效 |
| 1.6 | `stop_record` 末尾 `_cleanup_scrcpy_processes()` 强杀其它 scrcpy | `common.py` | 可能与投屏进程冲突；是否在 finalize 前误杀**未完全排除** | ⚠️ 谨慎：exclude_pid 在 record 已退出后无效 |
| 1.7 | `File.clear_file()` 内调用 `Scrcpy.stop_record()` | `common.py` | 录屏刚开始数秒被误停 | ❌ **已移除**；不要在清理临时文件时 stop 录屏 |
| 1.8 | stderr 使用 `subprocess.PIPE` 且不读取 | `start_record` | 录 1～2 分钟后保存**一直转圈**；日志停在 `stop scrcpy record process` | ❌ **stderr 死锁**（~64KB 满） |
| 1.9 | stderr 重定向到 `report/scrcpy_record.log` | `start_record` | 保存不再无限卡死 | ✅ **必须保留** |
| 1.10 | 改回 MP4 + 仍用 CTRL_BREAK 停止 | `start_record` / `stop_record` | 用户：**改回 mp4 后又出现同样问题**；`apm_2026-06-14-14-32-44` valid=False | ❌ 证明 **MP4 + CTRL_BREAK 不可行** |
| 1.11 | scrcpy + `CREATE_NEW_CONSOLE`（SW_MINIMIZE）+ `_windows_stop_scrcpy_console()`（AttachConsole + Ctrl+C） | `start_record` / `stop_record` | **脚本短测** 6s/45s：`valid=True`；用户：**网页全流程未确认稳定** | ⚠️ 当前代码方案；**不能以脚本为准** |

**重复犯错警示（MP4）：**

1. **不要**在 Windows 上对 scrcpy 录 MP4 使用 `CTRL_BREAK` 并假设成功（见 1.3、1.10）。
2. **不要**用「进程已 exit」或日志「finalized」代替 `_is_valid_record_file()` 检查。
3. **不要**对长时间录屏使用未消费的 `stderr=PIPE`（见 1.8）。
4. 无 `moov` 的 MP4 **无法事后修复**，只能重录。

## 1. MP4 保存失败 / 无法播放

### 1.1 用户可见现象

- 报告目录里**有** `record.mp4`（体积可达十几 MB～三十几 MB），但：
  - 报告列表**没有「播放」按钮**（`result.json` 里 `"video": 0`）
  - 浏览器 / 系统播放器提示**文件损坏**（Windows：`0xc00d36c4`）
- 保存报告时可能**长时间转圈**（30s+），或 Flask 进程卡死

### 1.2 后端判定逻辑

播放按钮是否出现，取决于文件是否通过校验：

- `File._is_valid_record_file(path, 'mp4')`：文件尾部 2MB 内必须能找到 **`moov`** atom
- `File.detect_record_video_flag()` / `resolve_record_video()` → 报告 API 的 `video` 字段
- 相关代码：`solox/public/common.py`（约 1384–1455 行）、`solox/view/apis.py`（`makeReport`、`/apm/record/*`）

**无 `moov` = 校验失败 = 无播放按钮 = 播放器报损坏。** 文件不是「没录到」，而是**未正常收尾**。

### 1.3 根因（已用日志/二进制分析证实）

#### 根因 A：scrcpy 在 Windows 上未「优雅停止」

MP4 的 `moov` 元数据在录制结束时写入。scrcpy 文档要求：**必须**用 Ctrl+C / 正常退出，不能强杀。

在 Windows + Flask 子进程场景下，以下方式**均不可靠或已证实失败**：

| 停止方式 | 现象 | 日志特征 |
|----------|------|----------|
| `proc.terminate()` / `kill()` | 进程立即退出，无 `moov` | `record not playable: valid=False` |
| `CREATE_NEW_PROCESS_GROUP` + `CTRL_BREAK_EVENT` | 进程 <1s 退出，仍无 `moov` | `record process exited after stop signal` 后 30s 仍 `valid=False` |
| `AttachConsole` + `GenerateConsoleCtrlEvent(0,0)`（**当前方案**） | 脚本短测 `valid=True` | `record process exited after Ctrl+C` + `record file ready` |

**失败案例（可复现分析）：**

- 报告：`report/apm_2026-06-14-13-37-11/`
  - `record.mp4` ≈ 34MB，有 `ftyp` + `mdat`，**完全没有 `moov`**
  - `result.json`: `"video": 0`
- 报告：`report/apm_2026-06-14-14-32-44/`（改回 MP4 后仍用错误停止信号时）
  - 14MB，`valid=False`，无播放按钮

#### 根因 B：stderr 管道死锁（导致保存卡死）

录屏进程若使用 `stderr=subprocess.PIPE` 且不读取，scrcpy 长时间写 stderr 会填满 ~64KB 缓冲区 → **子进程阻塞** → 停止信号无效 → 保存一直转圈。

**修复：** stderr 重定向到 `report/scrcpy_record.log`（见 `Scrcpy.start_record`）。

#### 根因 C：竞态 / 误杀（早期）

- `File.clear_file()` 曾调用 `Scrcpy.stop_record()`，可能在采集中误停录屏（已移除该调用）
- `makeReport` 与前端 `initializeAPM` 竞态：采集未就绪就 `record()`（前端已改为等 `/apm/initialize` 完成）

### 1.4 已尝试的修复路径

完整列表（含每次用户反馈与是否有效）见上文 **「修复方案尝试全记录」** 阶段 1～3。此处仅摘要：

1. 轮询 + 校验写回 `video`（必要但不充分）
2. ❌ Windows `CTRL_BREAK` 停 MP4（多次失败，**勿重复**）
3. ⚠️ MKV 兜底（能出文件，浏览器 seek 失败）

5. ⚠️ **当前**：scrcpy MP4 + 最小化控制台 + `AttachConsole` Ctrl+C（脚本通过，网页待验）

## 2. 画质极差

### 2.1 用户可见现象

- 选了 720p/1080p，画面仍糊，「连 480p 都不如」
- 与投屏画质选项无关时，录屏画质仍然很差

### 2.2 根因分析

#### 根因 A（主要）：使用 `adb shell screenrecord` 录 Cocos/Unity 游戏

2026-06-14 曾切换为设备端 `screenrecord` 以解决 MP4 finalize 问题，但：

- **`screenrecord` 录的是 Display 合成层**，对 **SurfaceView 游戏**支持很差
- 日志中游戏为：`SurfaceView - com.lyjz.chqsy.vivo/org.cocos2dx.cpp.AppActivity`
- 失败案例：`apm_2026-06-14-14-43-28`，文件 720×1560、15MB、**校验通过**，但画面仅为屏幕中央一小块模糊内容

**结论：MP4 有效 ≠ 游戏画面正确。**

#### 根因 B：scrcpy 录屏参数偏低 + 软件编码

当前录屏参数（`Scrcpy._get_cast_params(for_recording=True)`）：

| 档位 | max-size | 码率 | fps |
|------|----------|------|-----|
| 1080p | 1080 | 8M | 30 |
| 720p | 720 | 4M | 30 |
| 480p | 480 | 2M | 30 |

- 默认 `--video-encoder=c2.android.avc.encoder`（**软件编码**），避免高通 OMX 崩溃，但画质通常弱于硬件编码
- 投屏 + 录屏并行时，投屏被强制降到 720p/1M（**不影响录屏文件**，只影响 scrcpy 投屏窗口）

#### 根因 C：`--max-size` 语义

scrcpy 的 `--max-size` 限制**长边**像素。横屏游戏在竖屏手机上可能得到 **720×336** 一类分辨率（验收测试实测），属预期行为，不是播放器缩小。

### 2.3 当前方案

- **录屏**：scrcpy（能正确抓 SurfaceView）
- **画质档位**：首页「屏幕录制」→「录制画质」下拉（1080p/720p/480p），API 参数 `quality`

### 2.4 验收数据（2026-06-14，scrcpy 720p，45s）

- 路径：`report/acceptance_record_test/record.mp4`
- 分辨率：**720×336**（横屏游戏）
- 帧率：~30fps，时长 45.3s，9.73MB
- `valid_mp4=True`，中间帧截图 UI 清晰（见 `frame_mid.png`）

**接手者请用同一游戏做 A/B：screenrecord vs scrcpy，对比中间帧。**

## 3. 播放器中间只有一小块视频

### 3.1 用户可见现象

- 弹窗很大，视频只占中间小矩形，四周大黑边

### 3.2 根因 A：前端 CSS（竖屏视频）

旧样式（已改）：

```html

```

对 **720×1560 竖屏**视频：`max-height` 限制高度后，按宽高比算出的宽度约 **184px**，看起来「中间一条」。

### 3.2 根因 B：源视频内容就只有中间一块（screenrecord + 游戏）

### 3.3 浏览器播放 MP4 vs MKV

| 格式 | 浏览器 `<video>` | 进度条/时长 | 系统播放器 |

## 4. MKV 可以正常保存并播放

### 4.1 为何 MKV「看起来」更成功

- scrcpy `--record-format=mkv` 时，Matroska **头部在文件开头**，即使进程被强杀，文件也常「存在且有一定可播性」
- `File._is_valid_record_file(mkv)` 仅检查魔数 `\x1aE\xdf\xa3` + 大小 >2KB，**较 MP4 宽松**
- 因此 MKV 更容易出现 **播放按钮**（`video=1`），但不代表 finalize 完美

### 4.2 MKV 的问题（为何又改回 MP4）

- 浏览器内：**时长/seek 不可靠**（用户实测 00:00 / `--:--`）
- 与「报告内网页回放」产品目标冲突
- MP4 若在 Windows 上能可靠 finalize，体验优于 MKV

### 4.3 MKV 时期代码痕迹

- `resolve_record_video` 仍支持 `record.mkv`，`browser_playable=True`（曾改为 true）
- `wait_for_record_file` 同时轮询 mp4/mkv
- 测试：`tests/test_record_player.py`

### 4.4 若接手者采用「MKV 录制 + 转码 MP4」

推荐流程：

1. scrcpy `--record-format=mkv`（停止容忍度更高）
2. 停止成功后 `ffmpeg -i record.mkv -c copy record.mp4` 或重编码
3. 校验 `_is_valid_record_file(mp4)` 后再 `video=1`
